iOS Native-First Rebuild: Arkitektur

Tre approacher for att gora appen mer native. Klicka pa den du foredrar.

A

TabView + NavigationStack (Rekommenderad)

Ren SwiftUI-arkitektur. TabView med 5 flikar, NavigationStack per flik. Kalendern fixas med iOS 17 scrollTargetBehavior(.paging) istallet for UIPageViewController. WebView behalls bara for "Mer"-fliken.

Fordelar

  • Enklast -- minst kod, renast arkitektur
  • Ren SwiftUI = inga UIKit-bridging-buggar
  • iOS 17 scroll-APIer loser kalender-swipe nativt
  • Solo-developer-vanligt

Nackdelar

  • Horizontal scroll kraver tuning for oandlig-scroll-kansla
  • Deep linking mellan flikar kraver koordinering

Insats: ~2-3 veckor for kalender + kundlista + bokningslista

B

Coordinator Pattern + Repository Layer

Mer enterprise-aktig arkitektur med AppCoordinator, FeatureCoordinators, Repository-protokoll och Services. Skalbar men kraftigt mer boilerplate.

Fordelar

  • Ren separation av navigation fran vyer
  • Matchar webbappens DDD-Light-monster
  • Lattare att unit-testa navigeringslogik

Nackdelar

  • Mycket mer boilerplate (Coordinator + Repository + Service + ViewModel per feature)
  • Overkill for 3-5 native-skarmar
  • Webb-API:t AR redan service-lagret -- onodigt att duplicera

Insats: ~4-5 veckor

C

Inkrementell Hybrid (Bara fixa det som ar trasigt)

Behall WebView som primar. Fixa kalenderns swipe-bugg, lagg till kundlista som native tab. Minsta mojliga andring.

Fordelar

  • Lagst risk, snabbast att leverera
  • Kalender-fix pa dagar istallet for veckor
  • Bevarat monster (redan funkar for kalender)

Nackdelar

  • Fortfarande "webbig" for de flesta skarmar
  • URL-interception i WebView Coordinator blir rosslig
  • Vita blinkar och overgangar kvarstar for WebView-sidor

Insats: ~1 vecka
