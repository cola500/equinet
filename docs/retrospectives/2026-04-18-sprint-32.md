## Vad gick bra

1. **Feature Inventory sparade tid i S32-2.** Dev upptäckte att `BookingsListItem` redan täcker alla fält som detaljvyn behöver. Ingen ny API-endpoint behövdes, sparade ~2-3h API-arbete. "Kan detta lösas genom att ta bort kod?" fungerar också som "kan detta lösas utan ny kod?".

2. **Tre subagenter på S32-2 fångade olika problemlager.** ios-expert (plan): NavigationStack-struktur. tech-architect (plan): ViewModel-delning. code-reviewer (kod): force-unwrap, onSave-returvärde, duplicerad DateFormatter. Varje agent hittade distinkta problem -- minimal överlapp.

3. **S32-3 audit-först-approachen fungerade.** Planen började med att bygga tabell över 15 native-vyer × 4 kvaliteter. Audit-datan visade att ViewModels redan hade action-haptics, bara pull-to-refresh saknades i view-layer. Korrigerade scope innan implementation.

4. **Metrics-script levererade första dagen.** Shell-script + git log + grep räckte för baseline. Ingen Node.js-pipeline, ingen Vercel API, ingen dashboard. Enkelhet som funkar > komplexitet som lovar.

5. **Pre-commit-hooken (S31-4) fångade svenska-tecken-fel.** När tech lead (denna session) skrev backlog-rader fångades "pagaende" och "nar" och blockerade commit. Samma skydd för framtida sessioner. S31-4 validerad i praktiken.

## Vad som inte fungerade

1. **Parallell session krockade med delad working tree.** Dev 2 startade i huvudrepot efter dev 1 avslutats. När tech lead skulle uppdatera status.md pågick dev 2:s arbete på feature branch -- risk för krock. Löstes genom att skjuta upp tech lead-ändringar. **Beslut (från 2026-04-01) gäller fortfarande:** delad working directory = sekventiell, annars worktree.

2. **mobile-mcp visuell verifiering ej utförd i S32-2 eller S32-3.** Kräver bootad simulator + manuell interaktion. Verifiering skjuts till fysisk hardware inför lansering. Acceptabel avgränsning men betyder att vissa UX-aspekter (double-haptic på pull-to-refresh) inte är bevisade.

3. **S32-3 planfeltolkning.** Planen antog haptic saknades i Profile/Services/Announcements. Audit visade att ViewModels redan hade haptics. Hade planen kört `grep -r "UINotificationFeedbackGenerator"` i Steg 0 (aktualitetsverifiering) hade detta fångats. **Lärdom:** Aktualitetsverifiering-regeln måste gälla även audit-stories, inte bara "redan fixat"-fallet.

## Processändring till nästa sprint

1. **Aktualitetsverifiering ska täcka audit-stories också.** S32-3-planen antog gap som inte fanns. Lägg till i `docs/plans/TEMPLATE.md` att audit-stories kräver explicit grep-verifiering av påståenden om saknad funktionalitet.

2. **Sekventiella sessioner tills worktree-overhead är löst.** Delad working directory = koordinationsproblem. Om parallellt önskas: investera i dokumenterad worktree-setup som sparar tid snarare än ad-hoc-växlingar.

3. **Metrics-report vid varje sprint-avslut.** Lägg till i `.claude/rules/autonomous-sprint.md` Sprint-avslut som obligatoriskt steg. Ger historik vi kan jämföra mot.

4. **Patterns-uppdatering från S32-3 lärdomar.** Haptic-mönstret (ViewModel action-haptic + view refresh-haptic + `Bool.toggle()`-trigger) hör hemma i `ios-learnings.md` -- framtida iOS-sessioner ska slippa upprepa misstaget.
