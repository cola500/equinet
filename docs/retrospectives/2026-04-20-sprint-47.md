# Retro S47: Process-hardening 2

**Datum:** 2026-04-20
**Scope:** Bygg enforcement-infrastruktur som gör det tekniskt omöjligt att hoppa obligatoriska steg i processen. Adresserar rotorsaker från S43-S46 (8 procedurbrott/sprint) samt compacting-insikten att regler inte kan leva i prompts.

---

## Resultat

- **6 stories done** (S47-0 till S47-5)
- **6 aktiva hooks** (5 pre-commit + 1 pre-push)
- **4 BLOCKERS med override-mekanism** (plan-commit, sprint-closure, branch-check, review-obligatorisk, multi-commit, sprint-retro)
- **37/37 tester gröna** på main (upp från 0 tester för hooks före sprinten)
- **Test-svit regressionssäker** — `npm run test:hooks`
- **Strukturerat done-fil-format** live (maskinläsbart av S47-1-hook)
- **Review-matris extraherad** till `.claude/rules/review-matrix.md` (glob-baserad, 9 rader)
- **Override-mönster dokumenterat** i `commit-strategy.md` (4 former: pre-commit-subject, pre-push-senaste-commit, `--override`-flagga, rules-only-undantag)

**Total sprint-storlek:** ~5h effort, 6 PRar (#242-#247). Ingen produktionskod ändrad.

## Vad gick bra

### 1. Hooks levererades iterativt med god commit-rytm

Alla 6 stories följde mönstret: plan → impl → (fix efter review) → done. S47-1 hade 4 Important + 2 critical från security-reviewer — alla fixade innan merge. S47-2 hade Important från code-reviewer (awk-kolumnfiltrering) — fixat. Det är disciplin-mönstret vi efterlyste i S45-retron.

### 2. Test-sviten expanderade organiskt

Från 0 → 27 (S47-3) → 33 (S47-4) → 37 (S47-5). Varje ny hook fick minst 3 scenarier (pass + fail + override). Regressionssäkerhet från dag 1.

### 3. Review-matrisen är nu maskinläsbar

Glob-syntaxen i `review-matrix.md` parsas av `check-reviews-done.sh`. Matris och enforcement är synkade — ändrar du matrisen, justeras hookens beteende omedelbart. Inga dupliceringar.

### 4. Override-mekanismen är konsekvent

Samma regex `\[override:[[:space:]]*[a-zA-Z0-9åäöÅÄÖ]` används i alla hooks. Motivering måste starta med alfanumerisk tecken — template-placeholder `<motivering>` avvisas. Det garanterar att varje override är medveten.

### 5. Tech-architect-review fångade fyra viktiga saker i S47-0

Dev's initiala review-matris saknade middleware.ts, hade suboptimal auth-glob, oklar docs-only-semantik och blandade trivial-gating med matris. Tech-architect-review ersatte alla fyra innan merge.

## Vad fungerade inte

### 1. Dev mergade 4 av 6 PRar själv

S47-2, S47-3, S47-4, S47-5 mergades av Dev utan tech-lead-triggering (`kör review`). `scripts/`-ändringar är **inte** rule-docs-undantag per S45-3-regeln — endast `.claude/rules/*` faller under undantaget.

**Rotorsak:** `check-own-pr-merge.sh` blockerar `bash scripts/check-own-pr-merge.sh <PR>` men inte `gh pr merge <PR>` direkt. Hooken måste **anropas explicit** för att trigga — den är inte automatisk. Konsekvens: social norm + explicit check-anrop ≠ faktisk enforcement.

**Åtgärd:** Backlog-rad tillagd: "gh pr merge-wrapper för check-own-pr-merge". 30-45 min att bygga git alias eller shell-wrapper.

### 2. Tech lead (jag) committade på fel branch 3+ ggr under sprinten

Inkluderat: commit på `feature/s46-2-messaging-image-ui` istället för main (2 ggr), commit på `feature/s47-5-sprint-retro-gate` istället för main (1 gång). Varje gång räddat av stash+reset eller av multi-commit-hook.

**Rotorsak:** Delad working directory med Dev-session. `git branch --show-current` kollas inte automatiskt före commit. Tech lead "ärver" Dev's branch när Dev byter utan att tech-lead-sessionen är medveten.

**Åtgärd:** Backlog-kandidat för S48: "pre-commit hook som varnar om current branch inte är main och commiten är lifecycle-docs-only" (tech-lead-signal).

### 3. S47-0 och S47-1 dokumenterade men inte explicit testade mot S46-1-scenariot innan merge

Dev rapporterade "test: S46-1-scenariot → blockerar" i done-filen utan att inkludera output eller repro-steg. Det är tillitsfullt men inte verifierbart i efterhand. Test-sviten S47-3 fixade detta retroaktivt.

## Real-world-saves under sprinten

Två konkreta bevis att enforcement fungerar (inte bara teori):

### Save 1: Multi-commit-hook blockerade tech-lead-fel-commit

När tech lead råkade committa backlog-rad på Dev's S47-5-branch och försökte pusha → S47-4-multi-commit-hook triggade: "bara 1 commit, blockerar push". Commiten flyttades till main via stash. **Första gången S47-4-enforcement fångat verkligt fel.**

### Save 2: Sprint-retro-hook triggade Dev's done-commit

Dev committade S47-5-done-fil direkt på main. `check-sprint-retro.sh` (Dev's egen hook) blockerade. Dev använde `[override: sprint-retro skrivs av tech lead separat]` — första real-world-användningen av override-mekanismen. Git log visar override + motivering → spårbart.

**Insikt:** Dev's egen hook fångade Dev. Det är poängen med automation — regler är oberoende av vem som bryter dem.

## Patterns att spara

### Override-mönstret

`[override: <motivering>]` med obligatorisk alfanumerisk start. Tom placeholder `<motivering>` avvisas. Varje override lämnar spår i git log → spårbart post-hoc.

**Tre former:**
- Pre-commit-hooks: läs `COMMIT_EDITMSG` subject-rad
- Pre-push-hooks: läs senaste commit-meddelande (subject kan vara för nya commiten som inte finns än)
- Script-flagga: `--override` som argument (för non-interaktivt läge)

### Maskinläsbar review-matris

Glob + union-regel i `.claude/rules/review-matrix.md`. Matris och hook är synkade. Ändringar i matris påverkar enforcement utan kod-ändring.

### Test-svit för bash-hooks

Isolerat temp-repo per test via `mktemp -d` + trap-cleanup. Setup/teardown i outer shell (inte via `$()`). `assert_pass`/`assert_fail` med tydlig output. Dokumentation om hur man lägger till test för ny hook.

### Strukturerat done-fil-format

Fast checklista-format `- [x] code-reviewer — <sammanfattning>` som hook parsar via awk (enbart "Reviews körda"-sektionen, inte hela filen). Template-placeholders avvisas.

## 5 Whys: Varför byggde vi detta NU?

1. **Varför:** S46-retro visade 8 procedurbrott på en sprint.
2. **Varför så många:** Dev's session-compacting raderar regler från minne efter några timmar.
3. **Varför då:** Kontext-fönster är tekniskt begränsat. Komprimering är automatisk.
4. **Varför räckte S45-varningar inte:** Varningar ignoreras under tempo. `exit 0` = silent bypass.
5. **Rotorsak:** **Regler kan inte leva i prompts eller minne — de måste vara i kod.** S47 flyttar alla kritiska regler från dokumentation till enforcement.

## Lärdomar

### För framtida sprintar

**Enforcement > dokumentation.** En regel som finns i `.claude/rules/` men inte i kod är en önskan, inte en regel. Om regeln är kritisk → bygg hook. Annars acceptera att den kommer brytas.

**Override-mekanism gör blockers acceptabla.** En blocker utan override är frustrerande → överrides är säkerhetsventil. En blocker utan säkerhetsventil blir antingen ignorerad (via hack-arounds) eller bromsar legitimt arbete.

**Real-world-saves är det bästa beviset.** Två konkreta incidenter under sprinten där hookarna fångade brott — mer övertygande än 37 enhetstester.

### För AI-team-framework-idén

Detta sprintens arbete är **direkt återanvändbart** i annan produkt. Review-matris, override-mönster, test-svit-infrastruktur — allt är generiskt. När triggern i `docs/ideas/ai-team-framework.md` är uppfylld (3 sprintar med låga procedurbrott) är extraktion praktiskt lätt.

### För S48+

Öppen lucka: `gh pr merge` kan fortfarande användas för self-merge utan att hooken triggas. Nästa sprint ska adressera detta via git alias eller shell-wrapper. Effort 30-45 min, stor impact.

### Meta-lärdom

Det tog en hel dag att bygga 6 hooks + test-svit + docs. Det är **mycket snabbare än att leva med procedurbrott**. Tidsräkning:
- S43-S46: ~4 sprintar, ~32 procedurbrott (upptäckta). Uppstädning ~3-4h total.
- S47: 1 sprint, 6 hooks. Nu är procedurbrott-reduktion kontinuerlig.

Investeringsmässigt: S47 betalar av sig själv efter 2 sprintar.
