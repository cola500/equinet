## Vad gick bra

- **Session 1 (S27-1 till S27-5)** levererade 5 stories smidigt med TDD och code-reviewer
- **GDPR data retention** (S27-5) var den mest komplexa storyn -- Supabase Auth admin API, grace period, stale notification guard. Code-reviewer hittade 3 buggar som fixades innan merge
- **Verifieringsstories** (S27-6, S27-7) avslutades snabbt genom att bekräfta att arbetet redan var gjort
- **Parallella sessioner** (webb + ios/docs) fungerade utan merge-konflikter

## Vad som inte fungerade

- **Branch-förvirring i Session 2**: Checkout till feature branch fungerade inte som förväntat, commits hamnade på fel branch (S27-8). Cherry-pick löste det men kostade tid
- **S27-7 var redan klart**: Hjälpcentral native var fullständigt implementerad i en tidigare sprint. Feature inventory avslöjade detta, men storyn borde ha verifierats före sprintplanering
- **S27-6 och S27-7 var i praktiken verifieringsuppgifter**, inte implementation. Framtida sprintar bör skilja på "bygg" och "verifiera" i planering

## Processändring till nästa sprint

1. **Verifiera feature-existens i sprintplanering** -- kolla med grep/explore innan stories skrivs
2. **Branch-disciplin**: kör alltid `git branch --show-current` efter checkout, särskilt efter context-switch mellan sessioner
3. **Estimera verifieringsstories som 0.1 dag**, inte 0.5 dag
