## Vad gick bra

1. **Process-sprint med stort genomslag.** 7 stories på ~5 timmar total implementation — mest små, väl-avgränsade bash-scripts och prompts. Sonnet-hypotesen höll för hela sprinten.

2. **Bonus-hotfix bevisade att meta-processen fungerar.** Johans observation av "alla tjänster syns" ledde till defense-in-depth-commit (PR #207) inom 30 min. Snabbare återkoppling än sprint-cyklar.

3. **Self-testing-mönstret skalade.** Alla 7 stories följde samma mall: deklarativ regel → körbar gate. Ingen av dem krävde nya patterns eller djupa reviews.

4. **Alla nya hooks testades med positiva + negativa fall.** S36-3 (tech lead), S36-6 (slicing) och S36-0 (aktualitetsverifiering) har explicit test både att de varnar korrekt OCH att de inte falsk-varnar.

5. **M8 hittade rätt avvikelser.** S35-1 + S35-1.5 flaggades som modellval-fel — 100% precision på test-datasetet. Bekräftar att regeln är tillämplig.

## Vad som inte fungerade

1. **Tech lead bröt S33-0 Fas 4-regeln igen (men fångades av hooken).** När jag skulle skapa S36-3-story hamnade jag på dev:s S36-2-branch. Hooken från S36-3 gör att detta inte händer tyst mer — men själva hooken hade förhindrat min egen incident om den funnits dagen innan.

2. **Hotfix PR #207 var förhastad prio-bedömning.** Jag kategoriserade en seed-specifik auth-bug som "hög prio security" innan data. Faktisk verifiering (Provider-rad finns i DB, `getAuthUser` läser från DB) visade att koden filtrerar korrekt. PR #207 är fortfarande värdefull defense-in-depth men inte den blocker jag trodde.

3. **Seed-demo.ts data-läge okänt.** Lokal DB har 6 providers + 19 aktiva tjänster — mer än seed-test-users.ts skapar. Någon har kört seed-demo.ts. Inte ett sprint-problem men pekar på att seed-rutiner behöver dokumenteras bättre.

## Meta-observation -- self-testing-mönstret

Sprint 36 är första sprinten där **processen testar sig själv som struktur**, inte bara som enskilda punkter. Före S36: vi hade deklarativa regler (läs och kom ihåg). Efter S36: varje regel har minst en körbar gate eller metric som rapporterar avvikelser.

**Inventering efter S36:**

| Regel | Gate | Källa |
|-------|------|-------|
| Aktualitetsverifiering | Pre-commit hook blockerar plan utan sektion | S31-4 |
| Docs-matris | M7 post-merge compliance-check | S36-4 |
| Metrics vid sprint-avslut | Obligatoriskt steg i autonomous-sprint.md | S33-0 |
| Modellval | M8 avvikelse-larm | S36-5 |
| Arkitekturcoverage | TEMPLATE-sektion + subagent-prompt | S36-0 |
| "Vad reviewern missade" | Obligatoriska Täckning + Gap-sektioner | S36-1 |
| Tech lead-session | Pre-commit hook varnar vid feature branch | S36-3 |
| Seven Dimensions-slicing | Pre-commit hook varnar vid oslicat epic | S36-6 |

**8 regler med automatiserad uppföljning.** Innan sprinten: 2-3.

**Vad detta ENABLAR:** framtida arbete behöver inte oroa sig för att "komma ihåg" regler. Gate:arna fångar avvikelser när de sker. Retros blir mindre om att upptäcka, mer om att förbättra det som redan fångats.

**Risk att bevaka:** gate-trötthet. Om vi lägger till för många gates kommer varningar ignoreras. S36 avslutade med 3 nya (tech lead, slicing, compliance) + 2 rapporter (M7, M8). Det är nära taket. Nästa sprint ska inte lägga till fler — bara använda det vi har.
