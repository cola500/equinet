# Sprint 34 Retro -- iOS UX Major-fixar

## Levererat

| Story | Beskrivning | Modell | PR |
|-------|-------------|--------|-----|
| S34-1 | Profilvy: tap-targets (Byt bild, Redigera) + `.confirmationDialog` för Radera konto + accessibility | sonnet | #192 (964f20db) |
| S34-2 | Bokningsdetalj: mailto-länk, telefon+epost ≥44pt, `controlSize(.large)` + `role:.destructive` på Uteblev/Avboka, Label-knapp för hästnamn | sonnet | #193 (82f3892e) |
| S34-3 | Felmeddelanden: `LoginError`-enum med 4 kategorier + URLError/HTTP-mappning + differentierade meddelanden + ikoner i NativeLoginView | sonnet | #194 (411dfac0) |

**Totalt:** 3 stories, alla Major-fynd + 3 Minor-fynd från S33-1-audit åtgärdade. **1.5 dag effort som planerat.**

**Nya iOS-mönster etablerade:**
- `.confirmationDialog` för destruktiva actions
- `LoginError`-enum med message + icon som computed properties
- `accessibilityLabel` + `accessibilityHint` separation

## Vad gick bra

1. **Audit-till-fix-pipeline fungerade.** S33-1 genererade strukturerad fyndlista med kategoriserad allvar + mätvärden. S34:s stories var direkta översättningar av Major-fynden. Inga surprise-scope-changes.

2. **Subagenter fångade subtila problem.**
   - S34-2: code-reviewer hittade `.urlPathAllowed` fel charset för mailto (Major, fixat före merge)
   - S34-3: code-reviewer noterade att `.sessionMissing` och `.userBanned` mappades semantiskt fel (Minor, fixat)
   - S34-1: cx-ux-reviewer såg `accessibilityLabel`/`accessibilityHint`-blandning (Minor, fixat)
   
   Utan reviews hade dessa nått produktion.

3. **Sonnet + reviews levererade tre icke-triviala iOS-stories utan blockers.** Starkaste datapunkten hittills för modellval-hypotesen (se nedan).

4. **TDD i S34-3 fungerade.** 4 nya AuthManagerTests skrevs RED, implementation GREEN. URLError-mappning hade varit svår att få rätt utan test-first (URLError-catch FÖRE generic catch var icke-obvious).

5. **Inga merge-krockar mellan tech lead och dev.** Dev körde 3 stories utan att tech lead behövde röra working tree -- sprint-planeringen för S34+S35 var redan gjord i worktree före S34-start.

## Vad som inte fungerade

1. **"Redan fixat"-rate över mål fortsätter.** 5,2% post-S34 (mål: <5%). S33-retron flaggade detta men vi undersökte inte grep-pattern. Ska göras före S36.

2. **Docs-uppdatering skippades konsekvent.** Alla 3 stories har "Ingen docs-uppdatering (UI-only)". Tekniskt korrekt, men S34-1 introducerade `.confirmationDialog`-pattern och S34-3 etablerade error-enum-pattern -- borde ha lagts till i `ios-learnings.md` eller `patterns.md`. Flaggas som separat follow-up.

3. **Kända begränsningar lämnade odokumenterade.**
   - S34-3: `emailNotConfirmed` ger fortfarande "E-post eller lösenord stämmer inte" (fel meddelande). Noterat i done-fil, men ingen backlog-rad skapad.
   - S34-3: `.cancelled` URLError mappas till `.networkUnavailable` (app-navigering kan trigga detta felaktigt). Noterat men inte backlog-fört.

**Opus datapunkt:** S33-1 (visuell UX-audit över 15 vyer) -- också utan blocker.

**Hypotes validerad:** "Sonnet + disciplinerade reviews ≈ Opus utan reviews" håller. Vad vi SAKNAR data på:
- Säkerhetskritiska stories (RLS, auth-flöde) -- ingen sådan i senaste 10
- Komplexa arkitekturstories (nya domäner) -- S35-0 kommer ge första datapunkt
- Stories UTAN review (för att isolera review-effekten)

**Beslut för S35:** Fortsätt med Sonnet som default. Opus reserveras för **S35-0 (plan-review av Conversation-domän)** där arkitekturbeslut kan vara kritiska.

## Processändring till nästa sprint

1. **Grep-pattern för "redan fixat"-rate behöver granskas.** 2 sprintar i rad över mål. Antingen justera pattern i `scripts/generate-metrics.sh` eller acceptera ökningen som ny baseline. Undersök FÖRE S35-retro.

2. **ios-learnings.md + patterns.md ska uppdateras efter S34.** `.confirmationDialog`-pattern, `LoginError`-enum-pattern, `URLError`-catch-ordning, mailto-encoding-regel -- alla hör hemma där. Skapa som följande S34-uppföljningsstory eller inkludera i S35-0.

3. **Kända begränsningar → backlog-rader.** Done-filer är bra för lärdomar men begränsningar måste in i backlog för att inte försvinna. S34-3 hade 2 som borde till backloggen nu.

4. **S35-0 på Opus.** Första arkitekturella datapunkt i jämförelsen Opus vs Sonnet. Plan-review av Conversation-domän är kritisk (RLS, ownership guards, cross-domain-beroenden) -- rätt tillfälle att testa.
