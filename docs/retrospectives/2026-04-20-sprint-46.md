## Vad gick bra

### 1. Tech-lead-review fångade blockers innan de nådde prod

S46-0-planen granskades av tech-architect + security-reviewer. De identifierade tre framtida blockers (rate limit, magic bytes, transaktionellt upload-mönster) som dokumenterades som krav för S46-1. S46-1-review fångade att magic bytes-validering saknades. Allt fixat innan merge.

### 2. Magic bytes-validering — djupare säkerhet än planen krävde

Planen angav MIME-whitelist. Security-reviewer lyfte magic bytes (faktisk filinnehålls-validering, inte bara Content-Type). Implementerades i `supabase-storage.ts` med `file-type`-paketet. Skyddar mot MIME-spoofing där en angripare namnger en PHP-fil som `.jpg`.

### 3. iOS-auditen avslöjade pre-launch-blocker

S46-3 (iOS WebView-verifiering) hittade ett autentiseringsproblem som inte var känt: native login (Supabase Swift SDK) populerar inte WebView cookie-store. Ny enhet + native login → alla WebView-sidor (Meddelanden, etc.) visar fel. Utan auditen hade detta nått prod vid lansering. Backlog-item tillagt som pre-launch-blocker.

### 4. Process-automation testad i verklig sprint

S45 byggde hooks för plan-commit-gate, sprint-avslut-gate och docs-compliance. S46 var första feature-sprint med dem aktiva. Hookarna varnade korrekt men blockerade inte. Procedurbrott ≤2 (mål). Automationen fungerar.

## Vad kan förbättras

### 1. iOS live-test misslyckades — auth-flöde för QA ej dokumenterat

Live-test i simulatorn krävde inloggning mot QA. QA-auth-flödet (native login → session exchange → WebView cookie) är inte dokumenterat och visade sig vara brutet. Auditen fick genomföras som kodanalys istället.

**Prioritet:** HÖG — auth-desync är pre-launch-blocker. Granska `AuthManager.swift` och `native-session-exchange`-endpointen. Dokumentera korrekt QA-login-flöde i `ios-learnings.md`.

### 2. Ingen `capture`-attribut i fil-input

Sprint-spec markerade `capture="environment"` som "valfritt". Det implementerades inte. Leverantörer som vill fotografera hästar/skador direkt från kameran måste ta extra klick i fotopickern.

**Prioritet:** LÅG — backlogg-kandidat. Enkelt att lägga till efteråt.

### 3. S46-1 plan committades felaktigt på feature branch (inte main)

Commit-strategy säger att plan-filer (`docs/plans/*.md`) ska committas direkt på main, sedan feature branch skapas. S46-1-plan committades på feature branch, vilket skapade en onödig divergens vid merge. 5 Whys kördes i real-time (se retro-commit).

**Prioritet:** MEDEL — åtgärdat med 5 Whys. Mönstret ska bli automatik.

## Patterns att spara

### Transaktionellt upload-mönster (upload FÖRST, sedan createMessage)

För att undvika "meddelande utan bilaga" vid upload-fel: ladda upp till Storage FÖRST, få URL, skapa sedan Message med URL. Om upload failar skapas inget meddelande. Implementerat i `attachments/route.ts`.

```
1. Validera fil (MIME + magic bytes + storlek)
2. Sanitera filnamn
3. Upload till Supabase Storage
4. createMessage({ content, attachmentUrl, attachmentType, attachmentSize })
5. Return meddelande med signedUrl
```

### Magic bytes-validering med `file-type`

Lita aldrig på `Content-Type` header eller filnamnstillägg. Läs faktiska bytes:

```typescript
import { fileTypeFromBuffer } from 'file-type'
const bytes = await file.arrayBuffer()
const detected = await fileTypeFromBuffer(new Uint8Array(bytes))
if (!ALLOWED_MIME_TYPES.includes(detected?.mime)) throw new Error('Ogiltig filtyp')
```

Centralisera MIME-whitelist i `messaging-constants.ts`.

## 5 Whys

### Problem: iOS live-test kunde inte genomföras

1. **Varför?** Appen visade "Kunde inte ladda"-fel trots lyckad native login.
2. **Varför?** Native login (Supabase Swift SDK `signIn()`) sätter Supabase JWT i Keychain men startar inte MobileToken-exchange och populerar inte WebView cookie-store.
3. **Varför?** `AuthManager` anropar session-exchange conditionally — kanske inte triggades vid native login på QA.
4. **Varför?** QA-miljön (`equinet-app.vercel.app`) har aldrig testats för fresh-install + native login i simulator. Bara prod-instansen har testats manuellt.
5. **Varför?** Det saknas ett dokumenterat "QA fresh-install-testflöde" och ingen E2E-gate täcker auth-flödet på native login → WebView session.

**Åtgärd:** (1) Granska `AuthManager` — säkerställ att session-exchange sker efter `signIn()`. (2) Dokumentera QA-login-flöde i `ios-learnings.md`. (3) Lägg till backlog-item för native auth E2E-test.
**Status:** Backlog-item tillagt (pre-launch-blocker, hög prio).

## Lärandeeffekt

**Nyckelinsikt:** iOS-auditen S46-3 var värd sin vikt i guld — inte för att bekräfta att koden fungerar (det gör den) utan för att hitta en osynlig pre-launch-blocker i auth-flödet. Audit-stories bör alltid köras med LIVE-test som ambition, inte som dokumentationsövning. Om live-test inte kan genomföras är det ett fynd i sig.

**Till Dev (S47+):** Använd `[override: sprint-avslut]` i commit-meddelanden när du committar sprint-avslutsdocs (retro, status.md, done-filer) direkt på main. Signalerar att commiten är intentionell lifecycle-doc, inte ett processbrott. Hooken lär sig läsa taggen i S47.
