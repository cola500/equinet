# S51-0.1: MFA-verify rate-limit-ordning + IDOR-verifiering

## Aktualitet verifierad

**Kommandon körda:** Läste `src/app/api/admin/mfa/verify/route.ts` rad 30 — bekräftar att `rateLimiters.mfaVerify` anropas direkt efter auth-check, FÖRE zod och mfa.challenge. Grep efter `listFactors` i route-filen: saknas. Grep efter `challengeError`-audit-log: saknas `AdminAuditLog.create` i den pathen.

**Resultat:** Alla tre fynd bekräftade i aktuell kod (commit `515aabdd`).

**Beslut:** Fortsätt

## Bakgrund

Djävulens-advokat-review 2026-04-22 identifierade tre fynd i `POST /api/admin/mfa/verify`:

- **B1**: Rate-limit konsumeras på rad 30 (efter auth, FÖRE zod/challenge/verify). Konsekvens: legitim admin med nätverks-glitch kan bli utlåst, och en lyckad verify-success räknas som ett misslyckat försök.
- **B2**: Ingen ownership-check på `factorId` — `mfa.challenge({factorId})` anropas utan att verifiera att factorId tillhör sessionens användare. Defense-in-depth mot potentiell IDOR.
- **M1**: `challengeError`-pathen loggar warn men skriver inte `AdminAuditLog`, vilket gör brute-force mot borttagna/ogiltiga factors osynlig i forensics.

## Filer som ändras

- `src/app/api/admin/mfa/verify/route.ts` — ny ordning + IDOR-check + audit-log
- `src/app/api/admin/mfa/verify/route.test.ts` — 5 nya tester

## Approach

### Rate-limit-ordning (B1)

**Ny ordning:**
1. Auth check → 401/403
2. Parse JSON → 400 (rate limit ej anropat)
3. Zod-validering → 400 (rate limit ej anropat)
4. `rateLimiters.mfaVerify(user.id)` — pre-flight consume
   - Om blockad: 429 (utan ytterligare konsumering, sliding window konsumerar ej vid limit-nådd)
   - Om tillåten: token konsumerat, fortsätt
5. IDOR-check
6. Challenge → om fel: `resetRateLimit` (ge tillbaka token) + audit-log + 401
7. Verify → om fel: token stannar (konsumerat i steg 4) + 401
              om success: `resetRateLimit` + 200

**Netto:**
- Success: noll tokens konsumerade (consume + reset)
- Verify-fail: ett token konsumerat
- Zod-fail: noll tokens konsumerade (rate limit ej anropat)
- Challenge-fail: noll tokens konsumerade (consume + reset)

### IDOR-check (B2)

Efter Zod, före `mfa.challenge`: anropa `supabase.auth.mfa.listFactors()` och verifiera att `factorId` finns i `data.all`. Om inte: 403 + AdminAuditLog `"mfa.verify.idor_attempt"`.

### AdminAuditLog vid challenge-failure (M1)

I `challengeError`-pathen: lägg till `prisma.adminAuditLog.create` med action `"mfa.challenge.failure"`.

## Risker

- `resetRateLimit` på challenge-fail-pathen: om challenge-felet orsakas av nätverksproblem (inte admin-fel) är reset rimligt; om factorId är genuint ogiltigt ger vi tillbaka token — acceptabelt eftersom B2-IDOR-checken fångar ogiltiga factorIds dessförinnan
- Upstash sliding-window-semantik: `resetUsedTokens` nollställer hela fönstret. Vid success: admin kan omedelbart försöka igen. Acceptabelt för MFA.
